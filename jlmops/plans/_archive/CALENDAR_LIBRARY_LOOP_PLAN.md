# Calendar ↔ Library ↔ Task — Simplification Plan

**Status: All 6 phases complete and confirmed live (jlmops @460, 2026-07-08).** Phases 1/2/5 verified live earlier; Phases 3/4 confirmed live 2026-07-08 via a real end-to-end test (AYIW July email, calendar row → task chain → lazy entity → Doc attach). Phase 6 (docs) done the same day, plus a round of post-Phase-6 fixes also shipped and documented (title now sourced from the spawning task's `cal_Name` rather than derived from the slug; canonical concept-folder keeps the type prefix; calendar auto-sorts by date after a merge; `cal_Link` display simplified). Archived — every durable fact here has graduated into `WORKFLOWS.md` §13, `ARCHITECTURE.md`, and `DATA_MODEL.md`. The original investigation (below, still accurate as history) found that jlmops's content-publishing infrastructure — calendar, Library entities, task chains, translation — has a create path and a read path but no update/close-out path, so it constantly drifts from reality. Rather than building automation to keep it in sync, the agreed direction is the opposite: strip out the automation and machinery that requires syncing in the first place, and lean on the fact that a human (admin or manager) is already present at every real step.

## The three actors

- **Admin** (human) — drives the work, maintains the calendar directly.
- **Session** (Claude) — does everything mechanical the admin used to do by hand: drafting, image uploads, pushing to WordPress, building companion emails, calendar upkeep.
- **Manager** (human) — translates and edits in Google Docs, using Gemini for the actual translation pass. No session involvement in that step.

Any design choice below has to work for a human using a UI (admin, manager) and for a session acting through scripts/functions — these are different kinds of "user" and shouldn't be forced through the same interface.

## Verified breaks (original investigation, 2026-07-07 — still accurate evidence)

1. **Publish never closes the loop.** `blog-region-negev-en`/`-he` are `state: draft` in `SysLibrary` despite being live on jlmwines.com since 2026-07-06. `WebAppLibrary_markPublished` exists but nothing in the documented pipeline calls it.
2. **No target date → invisible, not just overdue.** Negev's entities have empty `slb_TargetDate`; per the old design's own rule, entities without one don't appear on the Calendar at all.
3. **Target date recorded as prose, not structured data.** `email-ayiw-2026-07-en/he`'s date lives only in `slb_Notes`, never in `slb_TargetDate`.
4. **Duplicate entities, conflicting dates, no de-dup.** Galilee had two entities (`blog-region-galilee-en`, `blog-galilee-en`) with different target dates, plus a third date from the user's planning sheet — three sources, no canonical answer.
5. **No update path once an entity exists.** Exhaustively checked `WebAppLibrary.js` — no function edits an existing `slb_TargetDate` or any other field generically after creation.
6. **No ingestion from the user's actual plan.** The user's Google Sheet (real forward plan) has no code path into `SysLibrary` — getting something registered requires a session to notice it and act by hand.

**Root cause, one sentence:** the system has a create path and a read/surface path, but no update or close-out path — so anything that changes after creation has no way back in except a raw spreadsheet edit that bypasses logging entirely.

**Decision made from this evidence:** don't build the missing update/sync path. Remove the parts of the system that need one.

## Corroborating code survey (2026-07-07, research fork)

A follow-up audit of the actual code against the simplification direction (not just the calendar) found:

- **`DATA_MODEL.md` is correct, `WORKFLOWS.md` is stale.** `slb_Version` is dead — `lockVersion` (`LibraryService.js:678-720`) doesn't touch `slb_State` or `slb_Version` at all, contradicting `WORKFLOWS.md` §13.3-13.4's description of a live version-bump step. The lock/version machinery this plan wants gone is already half-gone in code, just not in docs.
- **`attachExistingDoc`** already matches the target model — no fork/lock engine, just swaps the doc and archives the old one. Keep as-is once it gets the ownership-transfer fix (below).
- **`spawnContentChain`'s `isSiblingLanguage` branch** (`LibraryService.js:243-267`) is the actual auto-pairing mechanism — creates `baseSlug-en`/`baseSlug-he` together with cross-references in one call. This is what gets removed.
- **The entity drawer** (`PublishingView.html:414-420`) renders 7 header fields + 8 full sections for every entity regardless of type — Distribute, Family, Files & URLs, Attached tasks, References out, References in, State history, Activity log. Once auto-pairing and version history are gone, References out/in and State history have nothing left to show — concrete evidence behind "way more than I need."
- **Shared infrastructure, do not touch:** `TaskService.js` (`createTask`/`completeTask`), `TaskWidgets.html`/`ModalOverlay`, and `TaskPacks.html`'s non-content pack kinds (`deeplink_orders`, `deeplink_inventory`, etc.) are used by product-side task chains (product detail updates, counts, verification) too. Nothing here changes.
- **Resolved: `markPublished`'s `slba_ActionType='url-stamped'` fan-out** (`LibraryService.js:851-866`, flagged by the adversarial review as unaddressed) **is dead code once `slb_References` is dropped.** It iterates every entity looking for one whose `slb_References` names the just-published slug — with the field gone, this loop never matches anything. Remove the fan-out logic itself as part of this cleanup rather than leaving it running as inert code.

## The design

### Calendar

`JLMops_Publishing` (sheet ID `1l-mrCcmIYpkabTaxy4yTRmJ4nxqIaw7VwU2YRQ_8rbg`) becomes the one calendar — simple, directly editable, no derivation.

- **`refreshCalendarExport` retired** from daily housekeeping (`StatusReportService.js:610-688`, called from `HousekeepingService.js:734`). It currently wipes every non-holiday/blackout/note row daily and rebuilds from `SysLibrary` — exactly the automation being removed. Whatever's written to the sheet stays written.
- **`_loadHolidays()` (`WebAppPublishing.js:79-111`) loses its type filter.** Today it only returns `holiday`/`blackout`/`note` rows to the Calendar tab; it needs to return every row so entries the admin/session add directly actually show up in `PublishingView.html`'s Calendar tab.
- **Session-to-calendar write path: a pending-updates handoff, not a new credentialed script.** A session cannot write to `JLMops_Publishing` directly (verified — Drive MCP exposes read/copy/create-new-file only, no update-existing-file tool; jlmops's web app has no HTTP API either, `WebApp.js` only has `doGet`). Instead: a dedicated Drive folder (e.g. "Calendar Updates Pending") holds session-authored staging files (same `cal_Date`/`cal_Name`/`cal_Type`/`cal_Notes` shape as the real calendar) created via the session's existing `create_file` capability. A new jlmops function scans that folder, merges each file's rows into `JLMops_Publishing`, then archives the processed file (so it isn't reprocessed) — reusing the same `SpreadsheetApp.openById` access `_loadHolidays`/`refreshCalendarExport` already have, no new auth needed. This merge runs **both** ways: automatically as part of daily housekeeping (so staged updates land without anyone doing anything), and on-demand via a button in `PublishingView.html`'s Calendar tab (so the admin isn't stuck waiting for the next housekeeping cycle).
- **Merge is an upsert, keyed on `cal_Slug` always — no fallback matching.** Resolved (adversarial review finding #9): a `cal_Name` + `cal_Type` fallback isn't reliably unique (two different months' items could both be called "Newsletter"), so instead of a fallback, **`cal_Slug` is assigned immediately when a calendar row is first created — never deferred, never empty.** No content precedes the calendar: the slug exists from the row's very first moment, whether it's the admin typing a new row directly or a session staging one. This means the merge always has an unambiguous key to match on, with no window where a row exists without one. This is what prevents the exact "three conflicting dates for the same thing" mess found during the original investigation from recurring in the calendar itself going forward. Note this only prevents *future* calendar-row duplication — it does not retroactively fix the existing duplicate `SysLibrary` entities (see Open Questions below), which live in a different table entirely and need a separate one-time cleanup.
- **Admin retains independent direct-edit access** to `JLMops_Publishing` itself, by hand, anytime — entirely separate from the session's staging-folder path.
- **`cal_Slug` is the actual fix for duplicate entities, not a nice-to-have — but only if the bypass is also closed.** Root cause of the Galilee duplicate (`blog-region-galilee-en` vs. `blog-galilee-en`, two different slugs for the same real-world post): entity/task creation had no single authority to derive the slug from — the session and the admin each independently invented a slug for the same calendar item. `cal_Slug` fixes this only if there's no other way to create an entity. **Adversarial review (round 2) confirmed there currently is one:** all three live "create content" modals (`AdminTasksView.html`, `LibraryView.html`, `PublishingView.html`) take a free-text "Content name" field and derive a slug directly from it (`LibraryService.js`'s `_deriveBaseSlug`), with no calendar-row picker anywhere — exactly how `blog-galilee-en` was created originally. **The actual fix, confirmed correct:** those three modals' free-text name field is replaced entirely with a picker over *existing calendar rows* (same for the campaign field — pick from existing `SysMarketingCampaigns` rows, no free-text campaign name either). There is no path left to type a new name and get an independently-derived slug — the slug always comes from the calendar row picked, and (per the merge section above) that row already had its slug stamped the moment it was created, never invented by whoever's creating the entity later. Practical implication: creating genuinely new content becomes two steps — add it to the calendar first (fast, since the calendar is simple and directly editable, slug assigned right then), then start it by picking that row — not one step. This is a deliberate tradeoff for the single-authority guarantee, not an oversight.
- **`cal_Link`** (a plain URL to whatever's relevant — doc, live post, entity) stays a simple optional nice-to-have, unrelated to the slug-authority fix above.
- **Decided 2026-07-08: the Calendar tab shows a live, task-derived status per row, not just `cal_Notes` free text.** "Calendar is king" — it's the authoritative view of how an effort is going, so it shouldn't rely on someone remembering to type "published" into `cal_Notes`. Now that a calendar row can already resolve its matching entity (base-slug match added post-Phase-1, `PublishingView.html`'s `renderCalendar`), the same task-derived three-tier status defined in Drawers/UI above (not started / in progress / done) computes from that entity's open tasks and renders directly on the calendar row — no click-through required to see it. `cal_Notes` stays available for genuinely free-text notes, it just isn't the state signal anymore.
- **Decided 2026-07-08: a calendar row click routes to a task, not the entity drawer, when one qualifies.** Tasks tell the action story (Drawers/UI governing principle); the entity is a passive record. Role-aware, matching the existing Tasks-tab filter (`renderTasks()`'s `assignedTo !== 'Manager'` check): **admin** routes to the earliest-created open task for the row's matching entity, any assignee; **manager** routes to the earliest-created open task assigned specifically to `'Manager'` — a stage like Admin Review isn't theirs to act on, so it's skipped. Falls back to opening the entity drawer only when no qualifying open task exists (everything done, or nothing yet assigned to this role) — both roles already have Library tab access, so that's not a permissions gap.
- **Marking done = a direct edit** — by the admin directly on the sheet, or by the session via a staged update row applied through the same merge path above. No automated state transition derived from anything else in the system.
- **Cross-slug inputs (e.g., the newsletter pulling from that month's blog post and AYIW email) need no automated link either — same reasoning as dropping `slb_References`, extended across slugs, not just within one.** The newsletter is its own calendar row/slug/chain, separate from the blog's and the AYIW email's. Whoever works the newsletter task finds the current month's inputs by browsing the calendar (date-proximate, simple) and pulls in content directly (Drive/Library access) — exactly how the Print Newsletter Body section has always been hand-assembled from the locked blog body, never mechanically fetched. No structured "built from X + Y" field needed.

### Library entities

- **Consolidate the three duplicate "create content" implementations into one shared modal, first.** `AdminTasksView.html` (`createContentStream()`), `LibraryView.html` (`submitContentStream()`), and `PublishingView.html` (`submitContentStream()`) are three independently-built, hand-copied implementations — not one shared component (confirmed: `PublishingView.html:46`'s own comment says "same as LibraryView," and it only `include()`s `TaskWidgets`/`TaskPacks`/`TaskDetail`, never `LibraryView`). Extract one shared include (new file, e.g. `ContentStreamModal.html`, following the same `include()` pattern already used for `TaskWidgets`/`TaskPacks`), and have all three views call into it instead of maintaining their own copy. Build this ONE shared version as the calendar-row-picker (no free-text name/campaign fields, per the `cal_Slug` fix above) from the start — this also folds in `PublishingView.html`'s `spawnForEntity` and `AdminTasksView.html`'s `populateContentExistingEntities`, which are variations on the same duplicated pattern. Doing this first means the sibling-pairing removal (below) only needs to happen once, not three times. (Separate, not-yet-decided question this raises: `LibraryView.html` is still a live, independently-routed page — `WebApp.js:90`, `'Library': 'LibraryView'` — even though `PublishingView` was built to supersede it. Whether to retire that route entirely is bigger than this modal consolidation and isn't decided here.)
- **No automatic EN/HE sibling creation — everywhere, no exceptions.** Remove the `isSiblingLanguage` pairing branch in `spawnContentChain` entirely, including for `template`-type entities (welcome email, WhatsApp, pending-payment) which used the same mechanism. English and its translation are separate, independent entities, each created when the work for it actually starts — not batch-provisioned together, regardless of content type.
- **No lock/version/roll-forward machinery.** `attachExistingDoc` stays as the whole versioning story: attach a new version when there's one, replacing the old. Cut `lockVersion`'s realign-prompt branch (`peerNeedsRealignment`/`_flipPeerSlug` helper at `LibraryService.js:433`, `696-722`) — small, self-contained, low-risk.
- **Ownership-transfer fix generalizes to `attachExistingDoc`.** `createTranslationDraft` already does `copy.setOwner(adminEmail)` right after forking a doc (fixed 2026-07-06). `attachExistingDoc` never grants admin access — same gap, same fix, applied there too.
- **`slb_References` is dropped entirely, not kept as optional.** Verified (checked the only place it's ever settable — the free-text "References" field in the create-content modal, `PublishingView.html:454-455`/`LibraryView.html:282-283`, absent from `AdminTasksView.html` entirely) that it requires manually typing a known slug at creation time with no prompt or picker — confirmed unused in practice. Remove the field from the new consolidated create-content modal (see above); the column itself can sit dormant in the sheet (same as `slb_Version` — append-only schema rule, nothing physically removes a column, it's just never read or written going forward).

### Translation flow

The manager translates in Google Docs using Gemini. Two pieces of the current mechanism are good and stay; one piece is the actual hard link and changes; one piece gets removed entirely rather than fixed.

- **Keep as-is: the Doc-sourced Gemini prompt.** `_getTranslationPrompt()` (`LibraryService.js:1092`) reads a `template-xlt` library entity so the manager can refine the actual prompt wording in Docs, no deploy needed, with a sensible built-in fallback. `createTranslationDraft` prepends it to the top of the forked doc before the manager opens it. No change.
- **Ownership-transfer fix — kept, relocated.** `copy.setOwner(adminEmail)` still happens, but moved: Phase 3 generalized it into `attachExistingDoc` itself, so `createTranslationDraft` no longer sets it directly (would have been a redundant duplicate call on the same file). Same guarantee, one fewer place it could get missed.
- **✅ Shipped, revised same-day: the trigger changed, then changed again.** `createTranslationDraft` used to take `heEntityId` and derive the English peer via `_flipPeerSlug(heEntityId)` (slug-suffix math, now-deleted function) inside the translate task's pack. First cut moved the trigger to `task.content.edit`, gated on Done — reverted the same day (see Implementation sequence, Phase 4, for the full reasoning: it broke reachability and distorted the disable-on-Done convention). **Final: the trigger is back on the translate task**, but the EN peer lookup is now a plain deterministic slug flip (`this-slug-he` → `this-slug-en`, guaranteed correct because `spawnContentChain` names stage-resolved slugs that way from chain creation) rather than the old `_flipPeerSlug` function reaching for a possibly-nonexistent task. No Done-status check anywhere in the pack.
- **✅ Shipped: removed entirely, the "Open EN source" *link*.** `TaskPacks.html`'s separate "Open EN source" button and its old task-hunting visibility gate are gone — only the "Create translation text" *action* came back (on the translate task), not the navigation link. The manager already has direct Drive, Library, and Task access and can open the EN source themselves if they want to.

### Tasks

- **One chain per slug, not one chain per entity — entities attach to specific tasks progressively as the chain is worked, not all at once at creation.** (Corrected three times now — this is the settled model, shipped Phase 3.) Chain-creation itself is KEPT: `spawnContentChain` still creates one full ordered list of tasks for a slug up front (Create WP Stubs → Draft → Admin Review → Edit → Translate → Translate Edit → Images → Blog Publish, `WORKFLOWS.md` §13.2), matching "task chains are fine as a mechanism" (confirmed — product-side chains work well and aren't in question). What's removed is entities being force-created in pairs at that same moment — `spawnContentChain` creates **zero** entity rows now, not even one. **The concrete mechanism (settled during Phase 3 build, was ambiguous before):** each task already carries its stage-resolved slug (`baseSlug-en` for `en`-targeted stages, `baseSlug-he` for `he`-targeted ones) as `entityId` from creation — that part is unchanged. What's new is `createBlankDoc`/`attachExistingDoc` (`LibraryService.js`) auto-creating the entity, deriving type/language/title from the slug, the first time either is called against a slug with no entity yet. So the EN entity comes into being whenever the first task in the chain is actually worked (attaching/creating its Doc) — not deferred further than that, since there's nothing productive to defer it to — and the HE entity comes into being when Phase 4's translation trigger runs. By the time a downstream task (Images, Blog Publish) is reached, both entities already exist from earlier in the sequence — it just references them, with no ambiguity about which language a combined task belongs to. **This resolves the earlier "8 stages don't split per-entity" concern** — there was never a need to split the chain into an EN-chain and an HE-chain; it's one ordered list the whole time, and "one task = one attachment" describes when each task's entity gets created and filled in, not a requirement that the chain itself be divided.
- **Note (adversarial review round 2 correction, still applies):** the above does NOT resolve the Deficiency preset's "stalled" mislabeling — that mislabeling is unrelated to pairing (see Drawers/UI below for the actual fix, which drops the label instead).
- **Resolved: the progressive-attachment mechanism this requires didn't actually exist — now it does, with a minimal fix, not a redesign.** Red-team review (on the implementation sequence) found `TaskService.createTask` only ever sets a task's entity attachment at creation time — there's no update path for it, and the generic task-update endpoint's field allow-list doesn't include it either. Building "attach later, once work reaches that stage" as literally described would have required either modifying shared `TaskService.js` (explicitly off-limits) or a second, parallel task-writer (against the project's own conventions). **The actual fix, verified:** `WebAppTasks_updateTask` (`WebAppTasks.js`) already updates one other entity-linking field after creation — `st_LinkedEntityId`, confirmed working (`WebAppTasks.js:450-452`), just via a legacy field the modern Library integration doesn't read. Add `st_EntityId`/`st_EntityType` to that same function's existing allow-list, the same way `st_LinkedEntityId` already works. This is a two-field addition to an already-general-purpose update function — not new shared-infrastructure logic, not a parallel writer — and it makes progressive attachment work exactly as designed above.

### Drawers / UI

**Governing principle, decided 2026-07-08: "tasks tell the action story, library holds entities, calendar is king."** The entity drawer should show identity (title/slug/type/language/campaign) and actions (Open Doc, Attach new version, Create Content Tasks) and nothing else derived or historical — no computed groupings, no state pills, no audit trail. The one legitimate cross-reference is Attached tasks, because tasks are where the actual work and its status live. The calendar is the authoritative "how's this effort going" view, not the entity.

- **The Deficiency preset's "stalled" label is dropped entirely, not fixed.** Verified (`PublishingView.html:1265`, `LibraryView.html:552`) — `openCount === 0` currently renders "stalled" with no check against entity state, so a finished, *published* piece with zero open tasks gets mislabeled identically to an actually-abandoned one. Checked every use of the string "stalled" across the whole codebase: exactly these two spots, both pure display, nothing else reads it. No functional impact anywhere else, confirmed by the grep above, and dropping it removes a real false-positive instead of patching it.
- **Decided, refined 2026-07-08: replace `slb_State` everywhere (Library tab, entity drawer, and now the Calendar tab too) with one task-derived three-tier status, not a binary open/closed.** Verified `slb_State`'s actual write paths (`LibraryService.js`) — only three exist: entity creation sets `draft`, an explicit abandon action sets `abandoned`, `markPublished` sets `published` (requires a manual click in the Content-publish task pack, `TaskPacks.html:212` — nothing calls it automatically, the same root cause as Verified Break #1; Negev's entity is still `draft` despite being live since 2026-07-06, because it was published via `push-posts.js`/wp-admin directly, bypassing that pack). There is no writer for `locked` at all — `lockVersion` never touches `slb_State`, already noted in the Corroborating code survey above. So in practice every entity sits at `draft` forever unless someone explicitly abandons or manually publishes it — not real signal, stale by construction, the same failure mode as "stalled." The replacement, from task status (`New`/`Assigned`/`Review`/`Done`/`Closed`, `ADMIN_TASK_UI_PLAN.md`): **not started** (all open tasks are `New` — nothing picked up yet), **in progress** (at least one open task is `Assigned`/`Review` — someone's actively on it), **done** (every task `Done`/`Cancelled`, or no tasks ever spawned). One derivation, computed once, shown wherever a status is needed — Library rows, entity drawer, and the Calendar tab (see Calendar section above). Keep `abandoned` and `published` as separate, distinct flags — both are reliably set via a deliberate action (an explicit abandon, or the "Mark Published" pack click), not a drift-prone auto-transition; only the passive `draft`/`locked` default is the unreliable part. Confirmed still gating real logic, untouched: `PublishingView.html`/`LibraryView.html`'s `_renderDrawerActions` shows "Request Correction" only when `state === 'published'` and hides "Abandon" once `state` is `published`/`abandoned` — legitimate reads of the deliberate flags, not the derived display spectrum. **Implementation note, not yet audited:** check `renderDeficiency`'s preset logic for any `slb_State` dependency before removing the state filter — the Deficiency preset (pieces due soon/overdue) is target-date-driven per its description, but confirm it doesn't also gate on state before the field disappears out from under it. **Decided 2026-07-08: compute live, never persist back to `slb_State`.** Nothing outside the UI reads `slb_State` — no report, export, or automation depends on it — so there's no consumer a persisted value would serve that a live computation doesn't already cover, and persisting would mean building exactly the kind of write/sync path (hooking three separate task-write call sites in `WebAppTasks.js` to keep it current) this whole plan exists to eliminate. Live computation is always accurate by construction; a cached value can only ever be stale between writes. `slb_State` the column stays dormant for display purposes — the draft/locked/published *spectrum* is never shown again — but the two deliberate-flag reads noted above (`published`/`abandoned` gating the action buttons) are legitimate exceptions, not something to remove.
- **Decided: the "Family" section is dropped entirely, same as "stalled."** It's a computed convenience, not an action — the admin/manager already have direct Library access to find related pieces themselves. It was already broken by dropping slug-suffix pairing, and now that `slb_References` is dropped too (not kept as optional — see Library entities above), there's nothing left for it to group by anyway. No replacement needed.
- **Decided 2026-07-08: the "Files & URLs" section is dropped entirely, not trimmed.** Live-checked in the deployed drawer (`PublishingView.html:1417-1428`, chips for `docUrl`/`canvaDesignUrl`/`externalUrl`/`mdUrl`/`wpPostId`/`mailchimpCampaignId`) — in practice only the `docUrl` chip ever renders, and it's a pure duplicate of the "Open Doc" button already in the actions bar above (`:1442`, same link). The other fields (Canva, WP post ID, etc.) don't hold anything worth surfacing. Drop the whole section rather than keep a chip whose only live content is a duplicate button.
- **Decided 2026-07-08: the "Activity log" section is dropped entirely.** It's a raw, uncurated audit trail (`SysLibraryActivity` timestamp/action/summary/actor, `PublishingView.html:1353-1366`) — real forensic value but not something a person glances at to know what needs doing next, and per the governing principle above, tasks (not a parallel log) already tell that story. `logEntityActivity` keeps writing to the sheet either way (data isn't lost, just not surfaced in the drawer).

- `jlmops/docs/WORKFLOWS.md` — rewrite §13 (the 8-stage chain stays one ordered list per slug — describe progressive entity-attachment per task, not a per-entity split) and §13.6 (templates no longer sibling-paired). **Also** `jlmops/docs/ARCHITECTURE.md` and `jlmops/docs/DATA_MODEL.md` (found by red-team review — both also describe the old calendar regen-and-merge behavior as current).

**The pending-updates merge function and `ContentStreamModal.html` are the two genuinely new builds** — everything else on this list is removing, relocating, or minimally extending existing logic. None need new credentials: the merge reuses `SpreadsheetApp.openById` access jlmops already has, the modal reuses the existing `include()` pattern already used for `TaskWidgets`/`TaskPacks`, and the `WebAppTasks_updateTask` allow-list extension follows the exact pattern `st_LinkedEntityId` already uses in that same function.

Explicitly untouched: product-side task chains and the shared primitives they depend on (`TaskService.js`, `TaskWidgets.html`/`ModalOverlay`, `TaskPacks.html`'s non-content pack kinds).
