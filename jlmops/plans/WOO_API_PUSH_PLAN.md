Remaining items (category lookup, CSV export rework, push service) remain planned, not started — sequencing undecided (catalog/ops infrastructure, not one of the acquisition-period's two named channels; build now vs. queue behind acquisition-period work is an open call). Phase 1 of the bigger push/upsell initiative noted in `plans/STATUS.md`; upsell/cross-sell is a separate, later plan.
**Brand — dropped (decided 2026-07-20):** no brand data exists in current product data today, so leaving it out of the push is no loss. Google Shopping already maps Winery→Brand and JLM has high Shopping visibility on that mapping as-is, so there's no structured-data gap to fill by adding a native Brand field. The `SysLkp_Brands` build (former item 3) is cut entirely. The `Winery` custom attribute stays in the push, not pruned — kept as a hook for a possible future need even though it isn't used as a customer-facing filter today. **Note for that later plan (confirmed 2026-07-19 against WPML docs):** WooCommerce's `_upsell_ids`/`_crosssell_ids` are documented WPML "Copy"-type custom fields — set once on the EN product, auto-synced/ID-remapped to the HE translation, not independently authored per language. Matches the schema: `wpm_UpsellIds`/`wpm_CrosssellIds` exist only on `WebProdM` (EN); `WebXltM` (HE) has no equivalent columns. So the later plan should only need to read/push these on the EN side — confirm the live WPML custom-field setting is actually "Copy" before relying on it.

## Context

jlmops already pulls product descriptions and attributes from WooCommerce via REST during sync, but the *push* direction is manual today: someone generates an export file (new-product or updated-product) and copies/uploads it into WooCommerce by hand. This plan replaces the manual copy/upload step with an API push, using data already being generated today plus a few additional fields.

A real duplication bug in wine descriptions surfaced during discussion and was confirmed against both the live code and a live WooCommerce product export (`exchange/wc-product-export-16-7-2026-*.csv`): the description's detail-paragraph section repeats itself -- e.g. `Intensity: High (4 of 5) - High (4 of 5) High-intensity wines are bold and expressive...`. `WooCommerceFormatter.js:340-376` builds its own `"${label} (${code} of 5)"` header and appends a `SysLkp_Texts` lookup blurb (`getLookupText`) that *also* opens with the same label/value -- the blurb already carries the full "rating + prose" in one piece, so the fix is to drop the redundant header text and keep just `<strong>Intensity:</strong> ${text}`. Same pattern at Complexity (351-361) and Acidity (362-376). A byte-identical duplicate of this logic exists in a disabled dead-code branch (`USE_NEW_FORMATTING` is `true`; the `else` branch at 467-481/536-565 never runs) -- leave it alone, don't "fix" code that isn't live. The compact spec table above the prose (`col2Lines`, ~251-266, e.g. `Intensity: (four filled circles, one open) High`) is a separate, intentional summary and stays unchanged.

Bundles are explicitly **out of scope** (no jlmops authoring surface for bundle descriptions exists, and the owner considers the current manual bundle process fine). Non-wine category description backfill (gift/accessory/liqueur descriptions lost in a past bulk upload, restored manually on the live site but never back in jlmops) and the one-time cleanup of already-published duplicated descriptions are both explicitly manual/owner-handled, not part of this build.

## What an independent review confirmed and corrected

An independent agent re-verified this plan against the live codebase before build. Confirmed: the duplication diagnosis/fix above, and that `WooApiService.js`/`WooInventoryPushService.js` are solid reusable plumbing (generic REST `_fetch` with PUT/POST/retry, `SysEnv` auth; a working file-to-PUT template). It also surfaced real gaps, now resolved below:

- **No `POST`/create case needed.** Per the current, authoritative workflow (`jlmops/docs/WORKFLOWS.md` §14), the admin already manually pre-creates both the EN and HE WooCommerce drafts and enters the Woo Post ID into jlmops before a product ever reaches export -- so every push (new or updated product) is a `PUT` against an already-existing product ID. Simpler than originally framed; no create-path to build.
- **Category/attribute data does exist in jlmops** -- `WebProdM`/`WebXltM` (not `WebDetM`) carry `wpm_TaxProductCat`, `wpm_TaxProductBrand`, `wpm_AttrWinery`, `wpm_AttrIntensity`, `wpm_AttrComplexity`, `wpm_AttrAcidity`, populated from the existing Woo pull. `jlmops/docs/DATA_MODEL.md`'s `WebProdM` section was stale (documented 6 columns that didn't even match live field names; live schema has 66 + `wpm_ProfitRate` appended) -- **fixed 2026-07-20**, verified against `config/schemas.json` and a fresh `WebProdM` export.
- **Publish status is intentionally NOT part of the push payload.** The admin already manually creates the draft + translation at the start of the new-product process; ending the flow with an admin review-and-publish step is a **net workflow gain**, not a gap -- the push replaces the data-entry copy/paste, not the publish decision.
- **`manage_stock` will be stored as a real boolean in jlmops**, not the current `'yes'/'no'` string, avoiding a payload representation mismatch.

## Category and attribute-ID sourcing (settled after checking real data)

- **Category needs a WC term ID -- `SysLkp_Texts` is the right place to add it, not a separate table.** WooCommerce's `categories` field on product write requires `{id: <term_id>}` objects. `SysLkp_Texts` (`slt_Code, slt_TextEN, slt_TextHE, slt_Note` -- confirmed against real data, `exchange/JLMops_Data - SysLkp_Texts.csv`) turns out to already be the unified text/translation lookup `WooCommerceFormatter.js` reads at description-build time, and it already carries Category rows (Comax Hebrew category name to EN/HE, `Note=Category`) and ComaxCat rows (numeric Comax category code to name, `Note=ComaxCat`) -- i.e. it's already the Comax Division/Group <-> WC category join. Add a generic `slt_Value` column here (schema rule: append-only) rather than a category-specific column name or a separate table -- keeps it reusable for other future lookups. **`SysCategories`/`SysLkp_Texts` overlap resolved 2026-07-20 (owner confirmed, verified against fresh CSVs of both):** `SysCategories` (`sct_Code, sct_NameEn, sct_NameHe, sct_ComaxDiv, sct_ComaxGrp`) does track the same category/division data as `SysLkp_Texts`' `Category`/`ComaxCat` rows -- same ~6 wine categories keyed by `cpm_Group` Hebrew text (`sct_ComaxGrp`) and same Comax division codes (`sct_ComaxDiv`, including the 7/9→"Gift Items" double-mapping) -- but its data is stale/unmaintained, and it's read only by `ContactEnrichmentService.js` for CRM category-preference tagging. The two live production paths that actually need category data today -- packing slips and product descriptions (new + update) -- both read `SysLkp_Texts`, not `SysCategories`. So `SysLkp_Texts` stays the build target for `slt_Value`; consolidating onto `SysCategories` (which would need its stale data refreshed and its `sct_Code` slug-style values replaced with real WC term IDs) is a real future desire, not scheduled or decided.
- **Brand: dropped entirely, not pushed (decided 2026-07-20).** No brand data exists in current product data, so excluding it from the push is no loss. Google Shopping already maps Winery→Brand and JLM has high Shopping visibility on that mapping today -- there's no missing SEO/structured-data gap a native Brand field would fill. No new lookup, no schema changes, no admin UI card. (Superseded reasoning kept out of this doc; see git history if the earlier native-Brand approach needs to be revisited.)
- **Region and Grapes stay pruned; "Winery" attribute stays in the push, not pruned.** Region-as-WC-attribute would need accurate values re-supplied and doesn't actually help link region blog posts to products the way it might seem to. Grapes don't need attribute status either -- grape variety is already searchable another way. Both drop off naturally via WooCommerce's full-replace `attributes` array. Winery is different: kept as a hook for possible future use even though it isn't a customer-facing filter today (product-name search already surfaces most winery names; weak spot is a few generic words like "Zion"/"Jerusalem" that appear in multiple names). **Build implication:** because the `attributes` field is a full-replace array, the Intensity/Complexity/Acidity push (below) must also carry Winery's existing value in the same payload or it will get wiped when ICA is written. This is already available without new build -- `wpm_AttrWinery` on `WebProdM` is populated from the existing Woo pull -- so the export/push just needs to carry it through, not source it fresh.
- **Intensity/Complexity/Acidity don't need a lookup table.** These are just 1-5 numeric codes, already mapped to prose labels via `SysLkp_Texts` (`IN0X`/`CO0X`/`AC0X`). Their WC attribute *taxonomy* IDs (not per-value) are a small, stable set -- <=5 term values each, 3 groups -- a one-time lookup via `/wc/v3/products/attributes`, cached in `SysConfig` rather than a new table. WooCommerce's write API generally accepts plain name/value strings for global-attribute values directly -- to be confirmed with a real test push, not yet verified in this codebase.
