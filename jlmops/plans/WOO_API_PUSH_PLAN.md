# Woo API Push — Product Descriptions, Category, Brand, Attributes

**Created:** 2026-07-16/17 (scoped via live discussion, independent-agent review, and real sample-data checks against `WebDetM`/`WebProdM`/`WebXltM`/`SysLkp_Texts` and a live WooCommerce product export).
**Status:** Planned, not started. Sequencing undecided — this is catalog/ops infrastructure, not one of the acquisition-period's two named channels (content, offline); build now vs. queue behind acquisition-period work is an open call. Phase 1 of the bigger push/upsell initiative noted in `plans/STATUS.md`; upsell/cross-sell is a separate, later plan.

## Context

jlmops already pulls product descriptions and attributes from WooCommerce via REST during sync, but the *push* direction is manual today: someone generates an export file (new-product or updated-product) and copies/uploads it into WooCommerce by hand. This plan replaces the manual copy/upload step with an API push, using data already being generated today plus a few additional fields.

A real duplication bug in wine descriptions surfaced during discussion and was confirmed against both the live code and a live WooCommerce product export (`exchange/wc-product-export-16-7-2026-*.csv`): the description's detail-paragraph section repeats itself -- e.g. `Intensity: High (4 of 5) - High (4 of 5) High-intensity wines are bold and expressive...`. `WooCommerceFormatter.js:340-376` builds its own `"${label} (${code} of 5)"` header and appends a `SysLkp_Texts` lookup blurb (`getLookupText`) that *also* opens with the same label/value -- the blurb already carries the full "rating + prose" in one piece, so the fix is to drop the redundant header text and keep just `<strong>Intensity:</strong> ${text}`. Same pattern at Complexity (351-361) and Acidity (362-376). A byte-identical duplicate of this logic exists in a disabled dead-code branch (`USE_NEW_FORMATTING` is `true`; the `else` branch at 467-481/536-565 never runs) -- leave it alone, don't "fix" code that isn't live. The compact spec table above the prose (`col2Lines`, ~251-266, e.g. `Intensity: (four filled circles, one open) High`) is a separate, intentional summary and stays unchanged.

Bundles are explicitly **out of scope** (no jlmops authoring surface for bundle descriptions exists, and the owner considers the current manual bundle process fine). Non-wine category description backfill (gift/accessory/liqueur descriptions lost in a past bulk upload, restored manually on the live site but never back in jlmops) and the one-time cleanup of already-published duplicated descriptions are both explicitly manual/owner-handled, not part of this build.

## What an independent review confirmed and corrected

An independent agent re-verified this plan against the live codebase before build. Confirmed: the duplication diagnosis/fix above, and that `WooApiService.js`/`WooInventoryPushService.js` are solid reusable plumbing (generic REST `_fetch` with PUT/POST/retry, `SysEnv` auth; a working file-to-PUT template). It also surfaced real gaps, now resolved below:

- **No `POST`/create case needed.** Per the current, authoritative workflow (`jlmops/docs/WORKFLOWS.md` §14), the admin already manually pre-creates both the EN and HE WooCommerce drafts and enters the Woo Post ID into jlmops before a product ever reaches export -- so every push (new or updated product) is a `PUT` against an already-existing product ID. Simpler than originally framed; no create-path to build.
- **Category/attribute data does exist in jlmops** -- `WebProdM`/`WebXltM` (not `WebDetM`) carry `wpm_TaxProductCat`, `wpm_TaxProductBrand`, `wpm_AttrWinery`, `wpm_AttrIntensity`, `wpm_AttrComplexity`, `wpm_AttrAcidity`, populated from the existing Woo pull. `jlmops/docs/DATA_MODEL.md`'s `WebProdM` section is stale (documents 6 columns; the live schema has ~66) -- worth a docs fix separately, not part of this build.
- **Publish status is intentionally NOT part of the push payload.** The admin already manually creates the draft + translation at the start of the new-product process; ending the flow with an admin review-and-publish step is a **net workflow gain**, not a gap -- the push replaces the data-entry copy/paste, not the publish decision.
- **`manage_stock` will be stored as a real boolean in jlmops**, not the current `'yes'/'no'` string, avoiding a payload representation mismatch.

## Category, Brand, and attribute-ID sourcing (settled after checking real data)

- **Category needs a WC term ID -- `SysLkp_Texts` is the right place to add it, not a separate table.** WooCommerce's `categories` field on product write requires `{id: <term_id>}` objects. `SysLkp_Texts` (`slt_Code, slt_TextEN, slt_TextHE, slt_Note` -- confirmed against real data, `exchange/JLMops_Data - SysLkp_Texts.csv`) turns out to already be the unified text/translation lookup `WooCommerceFormatter.js` reads at description-build time, and it already carries Category rows (Comax Hebrew category name to EN/HE, `Note=Category`) and ComaxCat rows (numeric Comax category code to name, `Note=ComaxCat`) -- i.e. it's already the Comax Division/Group <-> WC category join. Add a WC-category-term-ID column here (schema rule: append-only) rather than standing up a separate table. (`SysCategories`, read elsewhere by `ContactEnrichmentService.js` for CRM purposes, may or may not track the same data -- open question, doesn't block this build.)
- **Brand moves to WooCommerce's native Brand field, not the "Winery" custom attribute.** Winery isn't actually used as a customer-facing filter today, and product-name search already surfaces most winery names well (weak spot: a few generic words like "Zion"/"Jerusalem" that appear in multiple names). Native Woo Brand is stronger for SEO/structured-data and Shopping-feed purposes, and jlmops already has an unused slot for it -- `wpm_TaxProductBrand` in `WebProdM` -- matching the already-known deferred item, formerly in `plans/STATUS.md` ("Woo Brand + GTIN structured-data enrichment... deferred alongside cross-sell"; that STATUS.md line now points here). Decision: **drop the "Winery" attribute from the push entirely** (it joins Region/Grape/Harmonize/Contrast on the dropped list -- see below) and push brand via the native Brand taxonomy instead.
- **Brand needs a new, dedicated lookup -- not `SysBrands`, not a product-pull capture.** Two corrections from checking real data: (1) `SysBrands` (`sbr_BrandEn/He`) is a dead end -- grepped it, only ever read by `ContactEnrichmentService.js` for CRM text translation, nothing writes to it or joins it to actual products. (2) Comax's `cpm_Vendor` is the *distributor*, not the brand -- not usable as a per-product brand source. A winery can also produce multiple brands (Golan Heights Winery -> both Yarden and Gamla), so brand isn't 1:1 with winery/producer. Decision: brand becomes a new, admin-maintained lookup, following the existing Grapes/Kashrut pattern (`WebAppLookups.js` -- a small allowlist `_WAL_ALLOWED_MAPS` of `map.grape_lookups`/`map.kashrut_lookups`/`map.text_lookups`, each backed by a `SysLkp_*` sheet and a card on `AdminProductsView`'s Lookups UI, shipped per the archived `LOOKUP_ADMIN_UI_PLAN.md`). Build: a new `SysLkp_Brands` sheet (key = brand name, plus a `Winery` column identifying the actual producing winery when it differs from the brand -- very few wineries have multiple labels, so mostly brand=winery, but the real relationship is captured when it's not) + a 4th allowed map + a new `wdm_Winery` column on `WebDetM` (admin picks the brand per product, same authoring shape as Intensity/Complexity/Acidity). Keying the table on brand (not winery) also means it naturally covers non-wine categories -- accessories have a brand but no winery, and the `Winery` grouping column is simply blank for those rows. The winery-grouping column itself is extra structure kept available for later use (e.g. winery-level content), not something the push needs today.
- **Region, Grapes, and the old "Winery" attribute stay pruned, not reinstated.** Considered and rejected: Region-as-WC-attribute would need accurate values re-supplied, but doesn't actually help link region blog posts to products the way it might seem to. Grapes don't need attribute status either -- grape variety is already searchable another way, so variety posts can already surface relevant products without it. All three stay out of the attribute push (they drop off naturally via WooCommerce's full-replace `attributes` array once brand moves to the native Brand field).
- **Intensity/Complexity/Acidity don't need a lookup table.** These are just 1-5 numeric codes, already mapped to prose labels via `SysLkp_Texts` (`IN0X`/`CO0X`/`AC0X`). Their WC attribute *taxonomy* IDs (not per-value) are a small, stable set -- <=5 term values each, 3 groups -- a one-time lookup via `/wc/v3/products/attributes`, cached in `SysConfig` rather than a new table. WooCommerce's write API generally accepts plain name/value strings for global-attribute values directly -- to be confirmed with a real test push, not yet verified in this codebase.
- **Winery-to-content linking (a blog post about a winery spanning multiple brands) is a known rough edge, not a blocker.** Auto-linking a winery-focused post to products via the brand field won't cleanly cover a multi-brand producer (needs per-brand logic or explicit per-brand mentions in the post) -- noted, but not a reason to change how brand is pushed.

## Scope (build)

1. **Fix the generator** -- `WooCommerceFormatter.js`: remove the duplicated label/value text from the prose-paragraph blocks (Intensity/Complexity/Acidity), keep the compact spec table unchanged.
2. **Extend `SysLkp_Texts`** with an appended WC-category-term-ID column, populated once via `/wc/v3/products/categories`.
3. **Build the new Brand lookup**: `SysLkp_Brands` sheet, 4th `WebAppLookups.js` allowed map, admin UI card entry (Grapes/Kashrut pattern), and a new `wdm_Winery` column on `WebDetM`. Populate the lookup's native-Brand term ID once the Brand taxonomy is in use.
4. **One-time attribute-taxonomy-ID lookup** for Intensity/Complexity/Acidity -- fetch `/wc/v3/products/attributes` once, cache the 3 groups' IDs in `SysConfig`.
5. **Rework the export generator** (`ProductService.js` -- `_buildProductDetailExport`/`generateNewProductExport`, currently builds a Google Sheet with only `WebDetM`+`CmxProdM` data) to: emit **CSV** instead of a Sheet, join in `WebProdM` (WC product ID) and `WebXltM` (HE translation product ID, keyed by `wxm_WpmlOriginalId`/`wxm_WpmlOriginalSku`), and look up category WC ID (`SysLkp_Texts`) and brand info (`SysLkp_Brands`, via the new `wdm_Winery`), plus manage-stock (boolean), qty, and Intensity/Complexity/Acidity (where relevant) in a fixed position order.
6. **New push service**, same shape as `WooInventoryPushService.js`: reads the CSV, `PUT`s to `/wc/v3/products/{id}` via `WooApiService.js` -- one call per product plus one for its HE translation row (description fields only for the translation call; category/brand/attributes/price/stock are shared, not per-language).
7. **Trigger**: on-demand, as a follow-on step after the existing add/update export is generated -- not a new standalone cadence.
