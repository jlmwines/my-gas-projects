## Punch list

### Priority 1 — measurable SEO impact

#### 1. Homepage hreflang URLs use `http://` — DEFERRED 2026-05-06

Both EN and HE homepages emit:
```html
<link rel="alternate" hreflang="en"        href="http://jlmwines.com/" />
<link rel="alternate" hreflang="he"        href="http://jlmwines.com/he/" />
<link rel="alternate" hreflang="x-default" href="http://jlmwines.com/" />
```

**Scope is homepage-only.** Posts, PDPs, and category pages all emit `https://` correctly.

**Root cause confirmed:** the unconventional homepage architecture (Settings → Reading = "Your latest posts" + `front-page.php` template-only override, no Page record anchoring the homepage URL). WPML's hreflang generation for the front-page URL takes a different code path than for normal Pages, and that path emits `http://` here.

**Fix path:** resolves as a side effect of Phase 1 of the homepage architecture rebuild — see `website/HOMEPAGE_BLOCKS_PLAN.md`. Once real Page records anchor `/` and `/he/`, hreflang generates from Page permalinks like every other URL on the site. No standalone fix attempted; deferred to that work.

#### 2. HE site name is `jlmwines.com` instead of `JLM Wines`

Title pattern across all HE pages:
- HE homepage: `jlmwines.com - מיטב היינות בישראל`
- HE post: `מתי כדאי להוציא יותר (ומתי לא) - jlmwines.com`
- HE PDP: `תבור הר רוזה - jlmwines.com`

EN equivalents end with `JLM Wines` consistently. RankMath title template uses `%title% %sep% %sitename%`. The `%sitename%` for HE resolves to `jlmwines.com` because WPML's per-language site name for HE is set to the bare domain.

**Fix path:** WPML → Languages → "Site name in this language" for Hebrew → set to `JLM Wines` (or `יינות JLM` if a transliteration is preferred). Re-fetch a HE page to verify.

#### 3. HE homepage URL in `page-sitemap.xml` is `/he/home-elegant/` instead of `/he/`

Sitemap entry submits the underlying canonical `https://jlmwines.com/he/home-elegant/` for the HE homepage. The actual user-facing URL is `https://jlmwines.com/he/`. The HE homepage HTML correctly canonicalizes to `/he/`, but the sitemap ships the deeper slug.

This causes Google to crawl two URLs (the slug and `/he/`) and decide which to index — usually it picks the canonical (`/he/`), but the lossy submission wastes crawl budget and adds duplicate-content noise.

**Fix path:** WPML → Languages → ensure HE language has a "Front Page" mapping that resolves to the HE homepage Page object, then RankMath emits `/he/` as the language homepage URL. Or set a per-page canonical override on the underlying `home-elegant` page so RankMath reports `/he/`.

User confirmed: categories ARE all translated. The audit's read of the sitemap was misleading — HE counterparts exist for all categories; the slug-pattern variation (some `-he` suffixed, some not) is what made the sitemap analysis look incomplete. No action needed.

### Priority 3 — opportunities

#### 9. Product schema lacks `gtin13`

Sample SKU on Tabor Har Rosé: `7290103681234`. This is an Israeli EAN-13 barcode (729 prefix = Israel GS1). Google rich results increasingly require GTIN to qualify for full Product rich result treatment.

**Fix path:** add a `gtin13` filter to RankMath's product schema in `seo-fixes.php`. For SKUs that are valid EAN-13 (regex `^\d{13}$`), emit them as `gtin13`. Keep `sku` as is.

```php
add_filter('rank_math/snippet/rich_snippet_product_entity', function ($entity, $product) {
    $sku = $product ? $product->get_sku() : '';
    if (preg_match('/^\d{13}$/', $sku)) {
        $entity['gtin13'] = $sku;
    }
    return $entity;
}, 10, 2);
```

Worth verifying which products have valid EAN-13 SKUs vs. internal codes before deploying.
